Single responsibility principle stands for A class should have only one reason to change .

The class responsibility depends on how the application changes.

If the application changes in ways that affect one of all the responsibilities, then the design will smell of Rigidity .
So we need to separate the class base on the responsibilities

In the other hand, the application changes in ways that not affect any responsibilities, we don't need to separate the class.
Indeed, separating them would smell of Needless Complexity .

The modem example (page 95) will describe detail of this principle.



Questions

Why does the class should have only one responsibility ?

When the requirements change, that change will be manifest through a change in responsibility amongst the classes

If a class has more than one responsibility, change a class for a responsibility may effect another. That make the design broken in unexpected changes

What is responsibility ?

In the context of the Single Responsibility Principle, the responsibility is a reason to change.
