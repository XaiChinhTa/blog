Liskov Substituable Principle

Liskov Substituable Principle is one of the prime enablers of the Open-Closed Principle .

Confirming LSP provides the substitutability of subtypes that allows a module, expressed in terms of a base type, to be extensible without modification


Unsubstituable

        class Shape() { 
            public get_shape() {
                // return shape
            }
        }
    
        class Circle extends Shape() {
            public draw() {
                // draw circle
            }
        }

        class Square extends Shape() {
            public draw() {
                // draw square
            }
        }
    
        drawShape(shape) {
            if (typeof shape == 'Square') {
                shape.cast_to(Square).draw();
            }
            else if (typeof shape == 'Circle') {
                shape.cast_to(Circle).draw();
            }
        }

With above design, Cirle and Square didn't override draw method from Shape.

Since Cirle and Square are not substitutable for Shape.

drawShape need to inspect its argument shape to detect the type. So this design violated LSP.

This violation forced the violation of OCP. Thus, a violation of LSP is a latent violation of OCP.


Principle Meaning

In example Square and Rectangle in page 113, we can see the substituable violation.

A model, viewed in isolation, cannot be meaningfully validated. It's only validated by the client after they use.

When considering whether a particular design is appropriate or not, we must view it in terms of the reasonable assumptions made by the users of that design. Often we will find that those reasonable assumptions are asserted in the unit tests written for the base class

But what assumption is reasonable? Indeed, if we tried to anticipate them, the design has the smell of Needless Complexity

Therefore, it is often best to defer all but the most obvious LSP violations until the related Fragility has been smelled.
